# Умный реестр подписок (AI-agent)

ReAct-агент на LangChain/LangGraph, который отвечает на вопросы о личных
подписках и платежах на естественном языке, вызывая инструменты для
получения данных и конвертации валют.

## Как запустить

### Вариант 1 — Docker

```bash
cp .env.example .env # предварительно вписать свой GROQ_API_KEY в .env
docker compose up --build
```

### Вариант 2 — локально

```bash
pip install -r requirements.txt
cp .env.example .env # предварительно вписать свой GROQ_API_KEY в .env
python main.py
```

### Тесты

```bash
pip install -r requirements.txt
pytest test_tools.py -v
```

## Какой LLM выбран и почему

**Groq (`openai/gpt-oss-20b` через Groq API)**.

- Бесплатный, с щедрыми лимитами и достаточно умными ответами.
- Очень низкая задержка инференса, что критично для агентного цикла:
  один вопрос пользователя может потребовать 2-4 последовательных вызова модели.
- Модель `gpt-oss-20b` достаточно хорошо следует инструкциям.
- Архитектура инструментов (`tools.py`) не завязана на конкретного
  провайдера — заменить `ChatGroq` на `ChatOpenAI`/`ChatAnthropic` можно,
  поменяв только конструктор `llm` в `agent.py`.

**Если облачная модель недоступна, то будет совершен переход на локальную `qwen3:14b`**
- Модель значительно умнее 7B, при этом не требует 40 ГБ, как 70B
- Нет зависимости от сторонних сервисов. Агент продолжает работать стабильно даже при сбоях облачных провайдеров
- Все данные пользователя остаются локально и не передаются в облако — это важно для финансовых сценариев
- Такая же легкая интеграция с LangChain
- Бесплатная



## Оптимизация и предотвращение зацикливания

На free-tier Groq ограничение 8000 TPM. Чтобы агент не зацикливался на множественных 
вызовах `convert_currency` (по одному на каждую подписку), в `tools.py` добавлен инструмент
**`calculate_total_spending()`** — он:

- Получает список подписок один раз
- Конвертирует каждую в целевую валюту внутри себя
- Возвращает готовый итог + breakdown по валютам

Это снижает количество LLM-циклов до одного вызова.
System prompt явно указывает агенту: «для вопросов "сколько всего" используй 
`calculate_total_spending`, а не цикл по convert_currency».

## Инструменты агента

**1. `calculate_total_spending(target_currency='RUB', category=None, status='active')`**
   - Быстрый способ получить общую сумму расходов в целевой валюте.
   - Внутри вызывает `get_obligations` и `convert_currency`, но агенту не нужно знать
     об этом — он просто получает итог.
   - Используется для вопросов типа "сколько я потрачу всего", "итого в рублях".

**2. `get_obligations(status=None, category=None, start_date=None, end_date=None)`**
   - Возвращает список обязательств (подписок/платежей) с фильтрацией.
   - Используется для детальных вопросов: "какие подписки есть", "какая категория самая дорогая".

**3. `convert_currency(amount, from_currency, to_currency)`**
   - Конвертирует отдельную сумму между валютами через frankfurter.app API.
   - Используется редко (обычно инкапсулирована в `calculate_total_spending`), но доступна для специальных случаев.

## Ограничения
- **Без долговременной памяти.**
  Каждый запуск создаёт новую сессию. 
  История предыдущих запусков не сохраняется.
- **LLM могла ошибаться в интерпретации дат периода.**
  Изначально расчёт границ "эта неделя"/"30 дней" был переложен на LLM через system prompt.
  Практика показала, что LLM могут некорректно вычислять границы относительных периодов
  (например, "эта неделя").
  **Это исправлено:** добавлен инструмент `resolve_date_range(period)`,
  который вычисляет границы в коде на Python, без участия LLM в арифметике.
  Агент теперь обязан вызывать его для любых периодов вместо счёта в уме.
  Текущая реализация поддерживает наиболее распространённые относительные
  периоды (`this_week`, `next_7_days`, `next_30_days`, `this_month`,
  `next_month`). Более сложные выражения естественного языка могут потребовать
  расширения сценариев использования.

## Мотивационное письмо
### 1
Мы тратим огромное количество ментального внимания на мелочи: не забыть отменить пробную подписку, проверить, не списалось ли лишнее, вспомнить, когда заканчивается страховка. Из-за этого обилия разрозненных подписок и платежей внимание рассеивается, и мы теряем фокус на действительно важных задачах.

Именно поэтому мне интересно создать универсального и надежного агента, который возьмет на себя рутину: соберет всю информацию о финансовых обязательствах воедино, быстро проанализирует ситуацию и выдаст четкий результат.

Такой агент не просто напоминает о платежах — он становится частью инструментария пользователя, помогая вернуть контроль над своими финансами и освободить ресурс для более важных решений.
### 2
Мне ближе роль продуктового разработчика, который смотрит на агента глазами пользователя.

Я бы хотела, чтобы конечный результат был действительно полезным и понятным.
Превращал сырые данные в чистый и ценный продукт. Чтобы не просто сдать проект, а сделать его таким, что он приносит пользу и мне.

Чтобы проект не остался просто галочкой в списке выполненных задач, а решал реальную проблему.
